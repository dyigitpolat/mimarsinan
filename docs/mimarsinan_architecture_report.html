Executive Summary

The system in three sentences

mimarsinan is a compiler from a declarative deployment configuration to a deployed, verified spiking neural network — it maps a trained ANN onto a neuromorphic chip, and the entire pipeline (steps, backends, mapping, training recipe) is derived from the config rather than hand-wired per feature. Its north star is to be generic, lossless, fast, and robust; concretely, deployed accuracy should match the ANN at a low spike budget (AC1: ≥96% @ S=4), stay lossless and monotone as the spike budget grows (AC2/AC3), each pipeline step should finish within a 5-minute wall budget (AC5), and nothing should crash (AC6). A large byte-identical structural refactor (V1–V8 + the Fix-A foundation E1–E6) turned the flag-thicket into a clean, axis-driven substrate but — by its own byte-identity discipline — moved no accuracy or speed numbers; the substrate was the prerequisite, not the cure.

The single headline change

Across all nine MNIST coverage templates, every fast/lossless lever was off — the cascaded-TTFS cell still ran the slow controller at its old-baseline accuracy. The Phase-2 work flipped the certified fast recipe on behind a per-cell deployed-accuracy + wall-clock regression gate (replacing byte-identity). Result: 8 of 9 cells certified PASS, the levered cells deploying at or above their floors while running 1.0×–1.7× faster; only matrix_3 (pruned LIF) regressed (−1.6pp) and was deliberately held on the controller.

Section 1

Original design intent — the north star

mimarsinan was conceived not as "a tool that supports many features" but as a compiler whose governing idea is that "the configuration is data and the pipeline is derived from it" — so the cross-product of deployment axes is covered without a combinatorial code explosion. Adding an option must cost ≈ one strategy class plus one registry entry; the blast radius of a change is the debt metric.

Generic

Any model / any dataset. Every cell of the firing × sync × encoding × mapping × backend × … cross-product must assemble. Adding an option = one class + one registry line (Principles 1, 2, 4).

Lossless

Deployed (on-chip-faithful) accuracy matches the ANN — AC1 ≥96% @ S=4, AC2 lossless at higher S, AC3 monotone in S. Measured on the deployed-forward metric, never a proxy.

Fast

Within a wall-clock budget — AC5 ≤5 min/step. The original slow controller ran 3–5× over budget.

Robust

AC6 zero crashes; validate up-front / fail loud (never core-dump); reproducible & pinned; every number is the deployed-forward, full-test, parity-gated number.

The 10 structural design principles (verbatim intent)DESIGN_GOALS §1
  1. Declarative → derived. The deployment config is the single input; everything downstream is resolved from it. Adding an option must not mean editing the pipeline — only registering a new strategy.
  2. Orthogonal composition (cross-product, not flag-thicket). firing × sync × encoding × mapping × backend × init × pruning × thresholding × spike-gen × bias × quantization × calibration × driver are independent concerns; any cell of the product must assemble. Two concerns interacting is a declared, tested compatibility rule — never a global if.
  3. One Source of Truth per concern. Each decision is resolved in exactly ONE place; the rest of the code reads the resolved decision, never the raw flags.
  4. Open–closed / bounded blast radius. Adding one option to one axis ≈ one new strategy class + one registry entry — not scattered edits. The blast radius IS the debt metric.
  5. Behavior-carrying polymorphism, not string-isolating predicates. A policy object carries the behavior so callers just call a method. Predicates isolate the symptom; policies cure it.
  6. Capabilities vs. strategies are separate layers. Capabilities = what the chip allows (declared); Strategy = how to fit a model onto a capability-gated chip (derived).
  7. Validate up-front, fail loud, never core-dump. Every backend×mode combination is checked at assembly against a capability matrix, not reactively mid-run (the SANA-FE SIGFPE core-dump is the cautionary tale).
  8. Equivalence-preserving evolution. Refactors are byte-identical (golden/parity/fidelity harness) or strictly better. The parity gates are the license to refactor aggressively.
  9. Reproducible & pinned. External deps are version-pinned; the deployment is deterministic given (config, seed, versions). Determinism is a feature, not an accident.
  10. The code reads like the spec. Abstractions mirror the domain axes; structural elegance is a maintainability goal, not a vanity one.

Source: docs/DESIGN_GOALS_and_refactoring_vectors.md §1.

The review's two-level verdict

The V1–V8 refactor must be judged on two charters. The structural charter (the 10 principles) is "a clear success, in places exemplary." But the behavioral program (the ACs that motivated the effort) was — by the byte-identity guarantee asserted in every commit — left essentially unchanged. A refactor byte-identical across a 7,680–15,360-config matrix cannot have moved a single accuracy or wall-clock number. AC6 (zero crashes) is the one genuine behavioral win.

Soft vs Hard Core Mapping, and the gate ladder (the deployed metric of record)

Soft Core Mapping (SCM) is hardware-shape-independent: it emits an IRGraph of logical NeuralCores + host ComputeOps, recomputes per_input_scales, quantizes the IR (if WQ), and runs the NF↔SCM per-neuron parity gate. It answers what to compute.

Hard Core Mapping (HCM) is hardware-shape-aware: it segments the IR at ComputeOp boundaries and packs soft cores into physical HardCores via best-fit greedy bin-packing — adding placement, padding, axon-coalescing, neuron-splitting, scheduling. It answers where to place.

The gate ladder:

  • Rung 1 — training/NF torch-forward accuracy (schedule-correct PyTorch).
  • Rung 2 — the SCM deployed metric: a 1:1 identity HybridHardCoreMapping (no pool/pad/reindex/coalesce/split) run through the genuine spiking forward. Isolates pure IR semantics.
  • Rung 3 — the HCM deployed metric: the same spiking forward over the packed mapping. Catches packing effects.
  • Rung 4 — backend parity (nevresim / Lava / SANA-FE vs the contract record).

The SCM "deployed metric of record" is evaluated on the FULL test set by default (deployment_metric_full_eval=True), the same test set as its torch reference — comparing a full-set NF number against a subsampled sim number would manufacture a spurious "NF↔SCM drop" that is pure sampling variance. The engine enforces a per-step performance tolerance: after each step, get_target_metric() ≥ previous_metric × step_tolerance (default retention 0.95), so the whole conversion is monotonically guarded.

Module dependency structure (one-way)

run.py → DeploymentPipeline → get_pipeline_step_specs (filtered _STEP_PLAN) + DeploymentPlan.resolve(config) → the engine runs each PipelineStep with a namespaced PipelineCache. Each step reaches into the domain packages: models/ (Perceptron, activations, SpikingHybridCoreFlow), tuning/ (AdaptationManager, tuners, OptimizationDriver, CalibrationPipeline), mapping/ (IRMapping→IRGraph, HybridHardCoreMapping, core_packing), transformations/ (weight quant, normalization-fusion math), chip_simulation/ (NevresimDriver, SpikingDeploymentContract, BACKEND_REGISTRY, SANA-FE/Lava runners), model_training/, data_handling/, search/.

Dependency direction is one-way: pipelining depends on all domain packages; the domain packages do not import pipelining. The DeploymentPlan + StepPlan + PipelineCache triad is the only cross-cutting infrastructure every step touches.

Section 4

What changed structurally — the refactor that made it generic

A pure-infrastructure, byte-identical program converted mimarsinan from a flag-thicketed pipeline into an axis-driven substrate. It came in two layers: the V1–V8 generic-refactor vectors (the substrate) and the Fix-A foundation E1–E6 (the seams the future behavior change "Fix B" plugs into). Every commit is locked byte-identical across the 7,680–15,360-config cross-product. The unifying reframing: the user declares a chip's capabilities and a workload; the framework derives the mapping/training strategy to fit one onto the other.

The Fix-A foundation (E1–E6)

E1 — Uniform rate-tuner seam (the three-verb contract)rate_tuner_seam.py

Fix A's real deliverable — "move the method up" was the easy 20%. Every rate tuner (LIF, the analytical clamp/shift/quant chain, the manager-rate tuners, the KD-blend tuners) now exposes the same three verbs so an OptimizationDriver drives any tuner generically:

  • ramp(rate) — predictor: apply transformation T at rate, return the post-apply metric.
  • recover_to(target) — corrector: fine-tune toward an accuracy target (routes the shared _recover_to_target SSOT; stashes the discovered LR on _last_recover_lr).
  • probe() — non-committing validation read.

The verbs delegate to each tuner's existing private methods — no behavior change. Framed as: the driver is an axis that drives an AdaptationAxis-shaped tuner; every tuner implements that shape.

E2 — Unbind OptimizationDriver into a pipeline-wide axisoptimization_driver.py + fast_ladder.py

The fast fixed-ladder driver was a KD-blend island; now every rate-tuner family (including the analytical chain and the manager-rate family, which had no fast path before) can be driven controller (default) or fast. Resolved by DeploymentPlan (plan.optimization_driver, plan.is_fast_driver). Default controller ⇒ fast ladder disabled ⇒ byte-identical. LIF stops bypassing the driver; the analytical chain finally gains a fast path.

E3 — Unbind CalibrationPipeline into a (firing × sync)-keyed axiscalibration_pipeline.py

The conversion-health steps (gain-correction / theta-cotrain / distmatch / boundary-STE) were resolved off ttfs_* flag names. E3 hoists the resolution onto the (firing × sync) contract: CalibrationPipeline.for_mode(...) reached via the contract. The ENABLE is the policy's does_conversion_health_calibration — True only for TtfsCascadeModePolicy; LIF/analytical/synchronized get inert(). All-off ⇒ byte-identical, but the analytical chain can now receive conversion-health steps the day a policy opts in.

E4 — The ConversionPolicy KEYSTONE (propose → confirm → escalate)conversion_policy.py

The keystone the action plan calls "precisely where Fix B turns into a silent-regression generator" if skipped. A static mode→recipe table is the right prior but unsafe alone: every proven recipe was validated on MNIST while the program promises "any model, any dataset." The three-stage layer:

  1. propose — propose_recipe(mode_policy) is the mode→recipe table (the prior). The cascaded fire-once cell carries train_s_hint=16, deploy_s_hint=32, staircase_ste=True, ste_mix=0.5; every other cell carries None hints.
  2. confirm — Characterizer.characterize(model, recipe) is the cheap pre-flight probe (cold_cascade_live / ramp_monotone / staircase_lif_ceiling / firing_gain; the real probes are research R1 — currently stubs). The default AlwaysMatchesCharacterizer keeps it inert.
  3. escalate — escalate_to_controller(recipe) rewrites a mismatched recipe to the controller fallback rather than shipping a silent regression.

DEFAULT-OFF, no-op: until conversion_policy is set, the decision names the CURRENT behavior (driver=controller) and the characterizer is never consulted ⇒ byte-identical. This is what makes Fix B safe to switch on later.

E5 — R6 deployment-faithfulness gatesdeployment_faithfulness.py

Institutionalizes "all accuracy is the deployed-forward, full-test, parity-gated number — the only number of record" (the saga had four "the torch metric lied" episodes). Three gate families: torch↔sim parity as a standing per-run gate; an external-dependency boundary audit (every optional dep — sanafe, lava, ffcv — is a declared integration boundary with a verify() guard); and drift detection (cross-checks the bootstrap_sanafe.sh pin against the code guard's _SUPPORTED_SANAFE_VERSIONS SSOT — a one-sided bump fails loud).

E6 — The certification protocol (REPLACES byte-identity as the Fix-B gate)certification.py

The protocol the review explicitly calls "the one you do not yet have." Byte-identity is the wrong gate for a deliberate behavior change. E6 is a frozen per-cell regression floor + a per-cell Pareto/regression gate on the deployed metric and wall-clock:

  • CertificationCell — canonical key mode[/schedule]@backend[#variant] (reuses the E3/E4 mode naming so a cell names the same thing across the program).
  • RegressionFloor — frozen {deployed_accuracy, wall_clock_s} + per-cell eps / wall_clock_budget_s + provenance (commit, timestamp, samples).
  • certify → one of three statuses: PASS (acc ≥ floor − eps AND wall ≤ budget), FAIL (names the regressing side), MISSING_FLOOR (no frozen floor — never a silent pass).

The transition protocol: build E6 → freeze the floor (a GPU matrix run) → swap the byte-identity gate for the per-cell Pareto gate → flip Fix B per cell, lowest-risk-first (LIF → cascaded → analytical → synchronized), each certified before the next.

The "consuming half" (EF1–3) + the per-layer-S Wizard axis (EW1–3)RESERVED / inert

EF1–3 wire the seams through the live run loop (still default-off ⇒ byte-identical): EF1 makes every single-switch family read the driver axis; EF2 is the E1↔E2 weld (the controller cycle routes its predictor/reads/corrector through ramp/probe/recover_to — "seam ≡ cycle"); EF3 consults the keystone in the live tuner-selection path, where an enabled decision's driver overrides the axis and its train_s_hint threads onto plan.train_s_hint.

EW1–3 reserve the per-layer-S axis (temporal_allocation.py) — the temporal analogue of mixed precision, where each cascade depth gets its own resolution S_d. The Wizard declares the intent (s_allocation ∈ {uniform | explicit | budget} + the allow_per_layer_s chip-capability gate); the per-depth S map is derived by the keystone (research, EW2) and certified by E6 (EW3). Default uniform repeats the global simulation_steps ⇒ byte-identical; nothing threads a non-uniform map into the forwards yet.

Why "the prerequisite, not the cure"

Grepping the 9 MNIST coverage templates for any fast/lossless lever (ttfs_theta_cotrain, ttfs_gain_correction*, ttfs_staircase_ste*, *_blend_fast*, fast_ladder, the genuine ramps) returned nothing. The cascaded-TTFS template still ran the slow controller at the ~0.26-cliff baseline. The templates demonstrate clean assembly / constructibility, not accuracy or speed. The byte-identity locks now defend the slow-controller-with-levers-off default — so turning the levers on will, by definition, break every lock. That transition is exactly the gap E6 fills.

Section 5

What changed behaviorally — the landings that moved numbers

Four landings actually moved deployed numbers. Three are Phase-1 deployment-fidelity fixes (U1/U2/U3); the fourth is Fix B, the first landing that deliberately breaks byte-identity — flipping the certified fast recipe on behind the per-cell deployed-metric gate.

U1 — Novena LIF gated behind a chip-faithful cycle-accurate forward

The LIF/Novena/offload/Loihi template (matrix_2) deployed at ~0.847 HCM — a multi-point fidelity hole below its near-ANN training accuracy. firing_mode selects the LIF reset rule: Default reset is subtractive (memb -= threshold, order-independent) while Novena reset is zero/hard (memb[fired]=0, arrival-order sensitive). The committed metric ran the analytical rate forward (constant per-cycle drive) but the deployed HCM cascades uniform spike trains — and because Novena is order-sensitive, that divergence (compounded by 5-bit weight quant) opened the fidelity gap.

U2 — per-source input scales applied for EVERY deployment, not only WQ runs

matrix_9 (ttfs vanilla, weight_quantization=false) tripped the NF↔SCM parity gate — a ~0.57 mismatch fraction against a 0.25 budget, a genuine deployment split. compute_per_source_scales (which bakes the upstream activation scale into the deployed weight via per_input_scales) was historically called only by WeightQuantizationStep — so when WQ was off, mapped weights shipped missing the input-scale factor.

U3 — decouple train_S from deploy_S + propose the R2 cascaded recipe (default-off)

R2 research found the genuine cascade fine-tune is S-negative (training the deep cascade at the high deploy-S collapses it — train-at-32 drops d9 to 0.770), while deployment wants the high quantization-floor S. U3 keeps _T = simulation_steps as the deploy-S (drives the deployed forward, node build, parity) and adds _train_T as the train-S. The ConversionPolicy proposes the R2-grounded cascaded recipe (train_s_hint=16, deploy_s_hint=32, staircase_ste=True, ste_mix=0.5). Default-off ⇒ _train_T == _T ⇒ byte-identical — this is the seam Fix B later pulls.

Fix B — the proven FAST recipe flipped ON in the certified templates

Fix B turns the certified levers on. Each flip adds two keys to the template ({lif,ttfs}_blend_fast + stabilize_steps:1200). This is the first landing that deliberately breaks byte-identity — per-cell certification on the deployed metric replaces it as the gate.

Controller driver (default) — robust but slow

The full SmoothAdaptation machinery: an adaptive rate scheduler + bisect, recover-to-target, rollback clone/restore, stabilization, a per-cycle LR find, a catastrophic gate, a target adjuster. High-quality but does many adaptation passes per cell — ~3.9–28 min/cell (matrix_7 235s → matrix_1 1703s).

Fast ladder driver — schedule, not search

Walks an explicit rate ladder (LIF default [0.25, 0.5, 0.75, 1.0]) with ONE shared optimizer + a spanning warmup→cosine LR, training a fixed steps_per_rate per rung — no per-cycle rollback / recovery / LR-find / stabilization. A final _fast_stabilize(steps) is a non-destructive bounded recovery on the deployed forward (rolls back on regression).

Section 7

Deep dive — the tuning / adaptation machinery

Every tuner is a rate tuner: it drives a single transformation T along a homotopy α-axis (α=0 = continuous/teacher model bit-exact, α=1 = fully-applied on-chip transform). The detail below is collapsible so the page stays scannable.

The tuner class hierarchy + the five mixins of SmoothAdaptationTuner

│   ├── AdaptationRateTuner (_apply_rate)  → ActivationQuantizationTuner, NoiseTuner

SmoothAdaptationTuner is composed from five mixins; the MRO is Seam → FastLadder → Cycle → Run → Base, so FastLadderMixin intercepts run()/_driver_attempt()/_stabilization_budget() and falls through to the controller path when the fast ladder is disabled.

The uniform RATE-TUNER SEAM (Fix A's real deliverable) — three verbs

Every rate tuner exposes a three-verb RateTunerSeam (a runtime_checkable Protocol) against which an OptimizationDriver is generic. The verbs delegate to existing private methods (byte-identical, no new control path):

VerbMeaningSmooth familyOne-shot

The key weld (EF2): the controller cycle now routes its predictor through ramp, its pre/post reads through probe, and its corrector through recover_to — "seam ≡ cycle."

Controller driver vs. Fast ladder driver (why the controller is slow)

The controller | fast decision is a single pipeline-wide axis (DeploymentPlan.optimization_driver). Default controller ⇒ byte-identical.

The controller is slow because the cost is dominated by repeated non-search work multiplied across many cycles: a per-cycle LR search (an exponential sweep, state cloned/restored after every probe), a clone/restore of full model state every cycle, up to 30 cycles (+ ~20 more bisecting toward 1.0), multiple validation passes per cycle, and a plateau-based recovery — then up to 3 LR-restart stabilization rounds at α=1.0. Empirically ~559s (LIF/MNIST).

The fast ladder walks the explicit rate list, commits each, never bisects; one shared optimizer + one spanning cosine LR; one probe read per rung; no rollback / LR search / clone-restore / recovery-to-target. A final _fast_stabilize is a bounded, rollback-guarded recovery on the deployed forward. Empirically ~60s (≈9× faster) matching the controller's deployed accuracy.

blend_fast mechanics + the stabilize_steps accuracy↔wall-time dial

blend_fast is the family of fast-ladder configs sharing proxy-fast warm-start across a rate ladder + a stabilize pass at the end:

  • LIF (lif_blend_fast): value-domain blend ramp over [0.25,0.5,0.75,1.0], steps_per_rate=120, eta_min_factor=0.1 (floors endpoint LR — the value-domain endpoint needs real LIF-dynamics training). _post_stabilization_hook → _fast_stabilize on the cycle-accurate deployed forward.
  • TTFS proxy (ttfs_blend_fast): the easier value-domain BlendActivation proxy ramp, then _fast_stabilize on the deployed _SegmentSpikeForward to close the proxy↔genuine cliff. Reaches a higher deployed accuracy than the genuine blend on MNIST.
  • TTFS genuine blend (ttfs_genuine_blend_fast): walks [0.5,0.75,0.9,0.97,1.0] training through a BlendedGenuineForward with CE(blend) + alpha·CE(genuine); _stabilization_budget()=0 (trains through the cascade for the whole ramp, cliff ≈ 0).

stabilize_steps is the clean accuracy-for-time dial. The ladder warm-start is cheap but the value-domain proxy endpoint isn't the deployed forward; _fast_stabilize(steps) is the only pass that trains the deployed dynamics, fresh-optimizer + spanning-cosine, rollback-guarded (restores if validation drops beyond tolerance). More steps = closes more of the proxy↔genuine cliff at linear wall-time cost; 0 skips it. It can never hurt (it is non-destructive).

The cascaded single-spike TTFS DEATH-CASCADE + the R2 STE-hedge fix

The problem. A cascaded ttfs_cycle_based model is a single-spike, ramp-integrate, fire-once cascade with a greedy partial-sum firing decision. At each successive depth the firing-gain is slightly deficient, so fewer/later neurons fire and the deficit compounds with depth — a death cascade that collapses to ~0.10 on a 9-deep model. The root cause is not quantization (LIF ≡ staircase bit-exact) and not a quadratic decode; it's a greedy partial-sum firing-gain deficit + a fire-once surrogate-gradient bug (training AT deploy-S with more budget hurts the deep cascade — the monotonicity invariant is violated). This caps deep cascaded genuine TTFS at a hard ~0.95 plateau.

The R2 fix — two independent levers (both default-off ⇒ byte-identical):

1. STE-hedge (ttfs_staircase_ste, ste_mix=0.5) — a straight-through estimator decouples forward from backward:

genuine   = fwd(x)                         # exact fire-once cascade (deploy path)
staircase = fwd._unpatched_forward(x)      # analytical complete-sum (cycle_accurate=False)
back = mix·staircase + (1-mix)·genuine     # hedged clean backward
ste  = back + (genuine - back).detach()    # FORWARD = genuine, GRADIENT = back

The forward is byte-identical to the deployed single-spike cascade (parity + deployed accuracy unchanged), but the gradient flows through a hedge of the clean complete-sum staircase gradient (no death-cascade pathology) and the genuine surrogate. The hedge is essential: pure staircase scrambles the genuine basin → chance; pure genuine stays on the plateau; mix≈0.5 escapes the plateau and reaches near-lossless deep cascaded TTFS in <2 min.

2. train_S / deploy_S decouple (R2's "two-residual" allocation): keep _T = deploy-S (deployed forward, node build, parity — all unchanged) and add _train_T = train-S. Hints: train_s_hint=16 (genuine-FT sweet spot — train-at-32 collapses d9 to 0.770), deploy_s_hint=32 (R1 staircase-ceiling S, capped because >S32 hurts d9). When train_S == deploy_S (default) the two builders are identical ⇒ byte-identical.

The S=4 caveat

At the matrix templates' deploy S=4, the controller's own floor is itself ~0.96 (death-cascade-capped at that S). So plain blend_fast already matches the controller floor at S=4 — the STE-hedge + S-decouple levers only pay off where the controller floor sits well above the fast-ladder result (deep cascades at higher S), not at S=4 where both are pinned by the same ceiling.

The calibration pipeline / distribution matching (the four conversion-health concerns)

CalibrationPipeline is the single declarative resolver of four conversion-health concerns (numerical ANN→SNN fidelity, largely orthogonal to the ramp), keyed (E3) by the (firing × sync) SpikingModePolicy:

  • gain-correction — invert the cascade's depth-attenuation. Two mutually exclusive modes: a COLD one-shot trim of each activation_scale by γ^d (ttfs_gain_correction), or a rate-gated RAMP (ttfs_gain_correction_ramp, wins when both set). The deployable "G" of the G→F recipe.
  • theta-cotrain (ttfs_theta_cotrain) — promote each channel's activation_scale to a per-output-channel requires_grad Parameter so the optimizer co-trains θ (firing-gain) per-neuron through the genuine cascade. The near-lossless cascaded recipe's key lever.
  • distmatch (DFQ) — match the deployed cascade's per-neuron channel-mean to the frozen teacher ANN's distribution (scale-aware [0,1] boundaries + DFQ per-neuron bias correction). Owned by the genuine-blend ramp. Revives death-cascade-collapsed neurons (cold 0.10 → 0.41).
  • boundary-STE (ttfs_boundary_surrogate) — un-sever the genuine backward at offload/host-ComputeOp boundaries so every segment trains on the deployed dynamics. Forward stays byte-identical; only the training gradient differs.

The ENABLE is mode_policy.does_conversion_health_calibration — today only the cascaded cycle opts in; LIF / analytical / synchronized get inert() = all-off = byte-identical. The parity-critical application order stays in the tuner (gain trim before node build; theta promotion after every scalar-θ calibration).

Cross-cutting infrastructure (sensors, recovery, checkpoints, budget)
  • AcceptanceSensor — pure, bit-exact extraction of the loop's statistical decisions: calibrate_baseline, absolute_floor (cumulative-drift guard), is_catastrophic, rollback_threshold, a paired-McNemar gate ((b10-b01)/N, tighter than the marginal 0.5/√n).
  • RecoveryEngine — the corrector primitive (train_to_target + recovery-hook remove-in-finally), with an optional persistent-optimizer so Adam moments survive the LR sweep within a cycle.
  • CheckpointGuard — scoped rollback snapshots; scope=tunable clones only requires_grad params, location=cpu_pinned does a real async pinned-host offload (~3–8× faster than pageable).
  • TuningBudget — all thresholds derived from check_interval = √SPE; accuracy_se() = 0.5/√(eval_sample_count) is the Bernoulli worst-case SE anchoring _rollback_tolerance = 3·SE.
  • AdaptationTargetAdjuster — proportional target decay (floor = 1 − degradation_tolerance), relaxed only on a 3-cycle stuck streak, clamped to the baseline-anchored floor.
