# chip_simulation/ -- Nevresim C++ Simulator Interface

Bridges Python and the nevresim C++ spiking neural network simulator.
Handles code generation, compilation, execution, and result parsing.
Also hosts optional Lava Loihi parity and SANA-FE detailed-stats backends.

## Key Components

| File | Symbols | Purpose |
|------|---------|---------|
| `nevresim/nevresim_driver.py` | `NevresimDriver` | Python-C++ bridge. Default **`connectivity_mode=runtime`** (`nevresim/connectivity.py`); override via config `nevresim_connectivity_mode`. Optional **`compile_cache_dir`**. |

| `nevresim/compile_nevresim.py` | `compile_simulator`, `CompileResult` | Compiles generated C++ with C++20; optional `-ftime-trace`, `-O0` |
| `nevresim/execute_nevresim.py` | `execute_simulator` | Runs binary; defaults to `cpu_count // 2` when `num_proc=0`. Drains stdout via `communicate()`, validates optional `expected_values`, surfaces stderr on failure. Protocol: one line per sample (`output_size` floats + newline). |
| `nevresim/segment_execute.py` | `run_binary_raw`, `save_segment_inputs` | SSOT for segment binary execution (input serialization, run, reshape). Used by `NevresimDriver` and hybrid precompiled path. |
| `nevresim/compile_cache.py` | `NevresimCompileCache` | Segment binary cache keyed by mapping + policy hash |

| `code_generation/main_cpp_template_runtime.py` | Runtime main template | ``RuntimeChip`` uses **static** storage (not stack) to support large core counts |
| `simulation_runner/core.py` | `SimulationRunner` | End-to-end simulation for single-segment and multi-segment (hybrid) mappings. Reads `nevresim_connectivity_mode` from pipeline config. Parallel emit+compile capped at `cpu_count // 2`. |
| `hybrid_execution.py` | `assemble_segment_input_*`, `apply_input_shifts_numpy`, `execute_compute_op_*`, … | Shared segment I/O and compute ops for hybrid backends. `apply_input_shifts_numpy` is the numpy mirror of HCM's `_apply_input_shifts` (negative-value shift before the encode clamp); ComputeOp execution runs in the op module's own floating dtype. |
| `hybrid_semantics.py` | `NeuralSegmentResult`, `store_neural_segment_output`, `lif_inter_stage_from_spike_counts` | **Inter-stage contract**: LIF/rate → spike count / T; TTFS → activation in [0, 1]. All hybrid backends must use `store_neural_segment_output` after neural stages. `lif_inter_stage_from_spike_counts` is re-exported from the SSOT `mimarsinan.spiking.segment_boundary.decode_segment_output` (single source across sims). |
| `ttfs_executor.py` | `TtfsAnalyticalExecutor`, `run_ttfs_hybrid_contract` | Canonical TTFS segment semantics and shared hybrid contract runner (numpy float64 compute ops). Applies `node_output_shifts` (negative-value shift) to each assembled neural-stage input; under the **synchronized** `ttfs_cycle_based` schedule it then snaps the stage input to the single-spike timing grid (`ttfs_input_grid_quantize`) — the spike-time-encoded hardware boundary cannot carry off-grid values. Used by `record_ttfs_hcm_reference` and SANA-FE `contract_ttfs_*` fields. |
| `ttfs_encoding.py` | `ttfs_spike_time`, `ttfs_single_spike_train`, `ttfs_latched_spike_train`, `ttfs_input_grid_quantize` | TTFS input spike encoding (`round(S·(1−clamp(x)))`), spike-train builders, and the encode→decode round-trip `ttfs_input_grid_quantize`. The scalar kernels delegate to the `models/spiking/wire_semantics.py` numpy twins (one source for torch and numpy). |
| `ttfs_segment.py` | `segment_ttfs_arrays_from_mapping`, `run_ttfs_*`, `gather_segment_ttfs_output_from_cores` | Numpy TTFS execution and output gather. |
| `ttfs_recorder.py` | `TtfsRunRecord`, `compare_ttfs_contract_records`, `compare_ttfs_hardware_records` | TTFS activation parity records; contract vs hardware tolerance policies. |
| `neural_segment_executor.py` | `execute_neural_segment_analytical` | TTFS analytical dispatch helper for shared hybrid loop. |
| `hybrid_stage_runner.py` | `run_hybrid_stages`, `HybridStageContext` | Shared stage loop with `on_neural` / `on_compute` (and optional `after_neural` / `after_compute`). Used by **nevresim** (`SimulationRunner._run_hybrid`), **SANA-FE**, **Lava**, and **`SpikingHybridCoreFlow`** (HCM). |
| `firing_strategy.py` | `FiringStrategy`, `FiringStrategyFactory` | Reset + threshold validation; backend capability gates; `require_chip_faithful_lif_forward(cycle_accurate_lif_forward)` — Novena's zero-reset is arrival-order sensitive so it requires the cycle-accurate cascade forward (the analytical rate forward deploys unfaithfully); `DeploymentPlan.resolve` enforces it. See `mapping/FIRING.md`. |
| `behavior_config.py` | `NeuralBehaviorConfig` | Identity axes of the simulator-facing activation semantics (reset, comparison, spike encode). Used by Lava, SANA-FE, and codegen helpers; composed into `SpikingDeploymentContract`. |
| `deployment_contract.py` | `SpikingDeploymentContract` | **Cross-side deployment-semantics SSOT** (torch NF + simulators): composes `NeuralBehaviorConfig` and adds `simulation_steps`, `ttfs_cycle_schedule`, `encoding_layer_placement`, `bias_mode`. **Single-reader invariant**: `from_pipeline_config` is the only place these config keys are read (grep-guard test); everything downstream takes the contract. Derived getters (`is_synchronized`, `quantize_stage_input_to_grid`, `training_forward_kind`) answer schedule-derived questions so consumers stop re-deriving implications; all accept a reserved `core=None` kwarg for future per-core heterogeneity. `mode_policy()` returns the (firing × sync) `SpikingModePolicy` (V2) that carries `training_forward_kind` and the soma/calibration behavior. `calibration_pipeline(config, *, distmatch_driven, core)` (E3) resolves the conversion-health `CalibrationPipeline` for this (firing × sync) cell — the pipeline-wide axis every conversion tuner consumes (cascaded cycle opts in; LIF/analytical/synchronized get the inert pipeline ⇒ byte-identical). `conversion_policy(config, *, model, characterizer, core)` (E4) exposes the characterization-and-policy keystone (`ConversionPolicy.resolve`, propose → confirm → escalate) — DEFAULT-OFF: until `conversion_policy` is set the `ConversionDecision` names the CURRENT behavior (driver=controller, no characterization run) ⇒ byte-identical. |
| `spiking_mode_policy.py` | `SpikingModePolicy`, `LifModePolicy`, `TtfsAnalyticalModePolicy`, `TtfsSyncCycleModePolicy`, `TtfsCascadeModePolicy`, `policy_for_spiking_mode`, `ExecPolicySpec`, `NevresimExecParams` | **Behavior-carrying policy per (firing × sync)** (Vector V2). One polymorphic class per `(spiking_mode, schedule)` family carries `training_forward_kind()` (the NF the fine-tuners train through), `calibration_forward()` (the negative-shift NF forward), `soma_hw_name()` / `soma_model_attributes(...)` (the SANA-FE soma chain), `log_potential`, `decode_mode()` (the count-vs-timing decode `models/spiking/hybrid/flow.py` routes its forward on), `nevresim_exec_policy(NevresimExecParams) -> ExecPolicySpec` (the **nevresim C++ codegen** firing × sync → `(ComputePolicy, Execution)` choice — `code_generation/generate_main.py:resolve_exec_policy` delegates here; the comparator/reset strings in `NevresimExecParams` are still selected by the codegen helpers, never here; the sync-cycle policy raises since nevresim never runs synchronized), `single_step_activation_replacement` (V5: whether one activation-replacement step subsumes the clamp/shift/quant chain — LIF + TTFS-cycle families; the step planner reads it via `DeploymentPlan.is_lif_style`), `does_conversion_health_calibration` (E3: whether this (firing × sync) cell runs the conversion-health calibration steps — `True` only for `TtfsCascadeModePolicy`, the key the `CalibrationPipeline` reads), and the **backend-capability SSOT** — `supports_backend(backend)` / `require_backend_supported(backend, context)` (single-backend gates) + `valid_backends(candidates)` (subset filter), all delegating to `spiking_semantics` as internals. `policy_for_spiking_mode(mode, schedule)` (= `from_contract`) is the SSOT dispatch; callers take the resolved policy instead of re-branching on `cascaded`/`synchronized` or calling `supports_spiking_mode`/`require_spiking_mode_supported` directly. Mirrors `spiking/segment_policies.py`. |
| `spiking_semantics.py` | `_BACKEND_CAPS`, `supports_spiking_mode`, `require_spiking_mode_supported`, spiking-mode taxonomy | **Per-backend capability matrix** + spiking-mode taxonomy (analytical/cycle/TTFS-family predicates). `_BACKEND_CAPS` declares which backends support which spiking modes. The capability functions are now the *internals* of `SpikingModePolicy` — call-sites (`behavior_config`, `sanafe/runner/core`, `backend`, `pipeline_helpers`) route through the policy rather than calling these directly. |
| `backend.py` | `Backend`, `SimulationBackend`, `BackendRegistry`, `BACKEND_REGISTRY` | **Backend interface + capability-validated registry** (Vector V3). `Backend.supports(contract)` / `require_supported(...)` consult the `_BACKEND_CAPS` matrix via `SpikingModePolicy.supports_backend` / `require_backend_supported` (V2: the policy is the capability SSOT); `SimulationBackend` keeps the existing simulation step classes (nevresim/loihi/sanafe) as the build+run+parity impls and carries each backend's `enable_*` predicate. `BACKEND_REGISTRY.selected_step_specs(plan)` validates every *enabled* backend×mode UP-FRONT at pipeline assembly (raising an actionable error before any step is appended) and returns its step specs — replacing the former informational matrix + reactive per-step validation. Lazy registry proxy avoids the chip_simulation←pipelining import cycle. |
| `certification.py` | `CertificationCell`, `RegressionFloor`, `CertificationFloorBook`, `CertificationStatus`, `CertificationVerdict`, `AbsoluteVerdict`, `certify`, `freeze_cell`, `floor_is_stale`, `load_floor_book`, `save_floor_book`, `FLOOR_BOOK_FORMAT_VERSION` | **Frontier E6 — the certification protocol that REPLACES byte-identity as the Fix-B gate** (pure data + freeze/gate functions; runs nothing). `CertificationCell` is the `(firing × sync × backend)` coordinate, canonical key `mode[/schedule]@backend` (reuses the E3/E4 mode naming; `from_mode_policy` builds it from a `SpikingModePolicy`), plus an optional `variant` discriminator (`mode[/schedule]@backend#variant`, default `None` ⇒ byte-identical key) that keeps distinct floors for deployment configs sharing one recipe cell (e.g. plain vs pruned LIF — the Phase-2 campaign proved one (firing × sync) cell has multiple floors). `RegressionFloor` is the frozen `{deployed_accuracy, wall_clock_s}` + per-cell `eps` / `wall_clock_slack` / `wall_clock_budget_s` + provenance; `CertificationFloorBook` is the JSON-serializable book (the frozen FORMAT, `format_version`-guarded). `freeze_cell` records the CURRENT numbers immutably (`scripts/freeze_certification_floor.py` is the CLI); `certify` is the GATE — PASS iff `deployed_accuracy >= floor − eps` AND `wall_clock_s <= budget`, FAIL naming the regressing side, MISSING_FLOOR (never a silent pass) for an unfrozen cell. **F1 absolute-AC overlay** (additive, byte-identical default): `RegressionFloor` carries optional `ac1_target` / `ac2_reference` (ANN ref, lossless) / `ac5_budget_s`, all default `None` ⇒ the floor round-trips byte-identically (`to_dict` omits unset overlay fields) and the relative verdict is UNCHANGED; never hard-fails). Byte-identical: nothing reads the floor until Fix B wires `certify` into the deployment gate. The floor is populated by a GPU matrix run BEFORE the first Fix-B flip (NOT this module). See `docs/CERTIFICATION_PROTOCOL.md`. Enforced by `tests/unit/chip_simulation/test_certification.py`. |

### Cross-backend numeric alignment

| Path | Dtype note |
|------|------------|
| HCM (`SpikingHybridCoreFlow`) | `torch.float64` default for membrane/TTFS math; weights float32 Parameters |
| TTFS reference | `TtfsAnalyticalExecutor` + `ttfs_segment.py` — single numpy semantics for HCM, Nevresim (raw activations), and SANA-FE buffer propagation |
| SANA-FE TTFS | Plugins + `TtfsAnalyticalExecutor` contract; hardware parity via `potential_trace`; no `sana_fe` core forks |
| Loihi/Lava | LIF only — TTFS does not map onto Loihi LIF dynamics; pipeline rejects `enable_loihi_simulation` + TTFS |
| SANA-FE (`SanafeRunner`) | `numpy.float64` segment assembly; float32 boundary drift can change ±1 spike at rate-encoding edges |
| nevresim TTFS | C++ `signal_t = double`; int rate-coded path uses exact integer arithmetic |
| Loihi parity | HCM `forward_with_recording` reference; per-core counts only at `B=1` |

All three deployment simulators depend on correct **`ChipLatency`** scheduling (see `mapping/ARCHITECTURE.md` § ChipLatency pitfalls) before parity gates are meaningful.

## Dependencies

- **Internal**: `code_generation`, `mapping`, `common`, `data_handling`.
- **External**: `subprocess`, `numpy`, `torch`; optional `lava`, `sanafe`, `ffcv` (runtime only). Each optional dependency is a declared integration boundary in `deployment_faithfulness.EXTERNAL_DEPENDENCY_BOUNDARIES` with a version-pin and/or capability-gate and/or lazy-import guard; the audit checklist (`test_deployment_faithfulness.py`) fails loud if a boundary lacks a guard.

## Dependents

- `pipelining.pipeline_steps.simulation_step` — `SimulationRunner` / nevresim.
- `pipelining.pipeline_steps.loihi_simulation_step` — `record_hcm_reference` + `LavaLoihiRunner` + `assert_spike_parity_or_raise`.
- `pipelining.pipeline_steps.sanafe_simulation_step` — `SanafeRunner` + optional HCM parity via `simulation_factory`.
- `pipelining.simulation_factory` — shared HCM build and metric test for SCM/HCM.
- `pipelining.core.pipelines.deployment_specs` — `BACKEND_REGISTRY.selected_step_specs(plan)` selects/validates backend steps at assembly.
- Entry point sets `NevresimDriver.nevresim_path`.
