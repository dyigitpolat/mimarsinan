# tuning/orchestration/ -- Smooth-adaptation orchestration loop + services

The control core of the tuning subsystem: the rate-search loop, the per-cycle
predictor→corrector→commit/rollback machinery, the adaptation-manager rate host,
and the decision services the refactor is extracting from the loop.

## Key Components

| File | Symbols | Purpose |
|------|---------|---------|
| `tuner_base.py` | `TunerBase`, `CATASTROPHIC_DROP_FACTOR`, `_RECOVERY_PATIENCE`, `_STUCK_STREAK_REQUIRED` | Shared infrastructure: trainer, budget, target adjuster, LR finder (`_find_lr`/`_get_cached_lr` with `anchor_lr=pipeline_lr`), `validate`. Tuning constants. `_create_trainer` builds the `BasicTrainer` with `recipe=self._tuning_recipe()` (the `tuning_recipe` block) and forwards `tuning_recipe_recovery` (config flag, default off → byte-identical) so the STEP recovery honors the recipe optimizer + warmup/cosine schedule instead of the hardcoded Adam/constant-LR path. |
| `smooth_adaptation_cycle.py` | `SmoothAdaptationCycleMixin` | One adaptation cycle, now **dissolved**: `_adaptation` is a 1-line delegate to `AdaptationDriver.run_cycle`, which drives the named phase methods here — `_begin_cycle` (clone pre-state + pre-cycle read **through the seam `probe` verb**), `_probe_instant` (predictor **through the seam `ramp` verb** + catastrophic gate), `_recover` (corrector **through the seam `recover_to` verb** — which routes the `_recover_to_target(target, rate)` SSOT and stashes the LR on `_last_recover_lr`, read back here for the trace; seam ≡ cycle, EF2), `_measure_post` (post read **through the seam `probe` verb** + dual rollback gate), `_rollback_cycle` / `_commit_cycle` (restore-or-commit + stuck-streak target relaxation). Records one `DecisionRecord` per exit into `self._cycle_log`. Accept/reject decisions delegate to `AcceptanceSensor` (static methods); `_absolute_post_acc_floor` delegates too. Under `tuning_use_paired_sensor`, `_measure_post` derives `post_acc` from the paired correctness vector itself (one eval pass, one statistical basis — D6); a non-negative `global_budget` is enforced at construction (D4). **Recovery-quality knobs** (all default-off → byte-identical): (1) `tuning_refind_lr_on_miss` invalidates the cached LR when a committed cycle misses the target (`_commit_cycle`) so the next cycle re-discovers it (else the LR is cached at cycle 0 forever); (2) `tuning_recovery_lr_plateau` (+`_factor`/`_reductions`, `_recovery_plateau_kwargs`) passes plateau LR-reduction kwargs to `_recover`'s `RecoveryEngine.train_to_target` (off → 1.0/0 no-op); (3) `tuning_rollback_ratchet` (+`tuning_rollback_cumulative_bound`, default 0.05) is the NON-STALLING ratchet in `_measure_post`: `AcceptanceSensor.ratchet_threshold` keeps the per-step relative gate (`pre_cycle_acc - margin`, so the ramp keeps climbing) but caps the cumulative drift below the `_best_committed_acc` high-water mark (`_ratchet_best_anchor`, tightened on each commit by `_update_best_committed_acc`) — repeated sub-margin slips cannot accumulate and the bound tightens as best ratchets up; (4) `tuning_tight_plateau` (+`tuning_recovery_check_divisor`, `_recovery_check_interval`) divides the recovery check interval so the stale-streak patience trips after fewer steps; (5) `tuning_stabilization_bounded` makes `_recover` request `return_steps` and `_accumulate_gradual_steps` sums them into `_gradual_train_steps` (the bounded-stabilization budget, consumed in `smooth_adaptation_run.py`). **CERTIFIED non-destructive keep-best** (`tuning_keepbest_certified`, R7d, default off → no snapshot taken): on each commit, `_snapshot_best_committed_if_improved` clones the model STATE into `_best_committed_state` (with its metric in `_best_committed_metric`) iff the commit's gate metric STRICTLY beats the running best — the run-level keep-best record distinct from the ratchet's scalar `_best_committed_acc` (which snapshots no state). The gate metric is `_certified_gate_metric`: the paired correctness fraction over the shared confirm subsample when `tuning_use_paired_sensor` is on (deployed-anchored where available, reusing the commit-time paired `post_acc`), else `probe()` — the SAME basis the commit gate uses. The finalize-time restore lives in `smooth_adaptation_run.py`. **GENUINE full-transform probe** (`tuning_full_transform_probe`, default off): after each commit, `_probe_full_transform` measures BOTH the value-domain (`_value_full_transform_eval`, always non-destructive clone→rate-1.0→restore) AND the genuine deployed-forward (`_full_transform_eval`, base default = value; overridden by `KDBlendAdaptationTuner`) rate-1.0 accuracy. Logs a per-commit record `{committed, committed_acc, value_full_acc, genuine_full_acc, value_drop, genuine_drop, proxy_gap}` (with `full_acc`/`drop` aliasing the value pair for the legacy test). `proxy_gap = value_full_acc − genuine_full_acc` is the value↔genuine divergence the value proxy hides. The verdict (`_log_full_transform_trend`, in `smooth_adaptation_run.py`) keys on `genuine_drop`. **AC5 per-fine-tuning-PASS wall (timing-only, byte-identical):** the corrector `_recover_to_target` and the below-floor recovery `_attempt_recovery_if_below_floor` time their `RecoveryEngine.train_to_target` call through the run-owned `FtPassWallLog` (`_ft_pass_wall_log().time_pass("recover" | "below_floor_recover")`); `max_ft_pass_wall_s` / `ft_pass_walls` / `ft_pass_wall_metrics()` surface the worst single FT pass + breakdown so AC5 is judged per FT step, not on the end-to-end wall. |
| `smooth_adaptation_run.py` | `SmoothAdaptationRunMixin` | `run()`: baseline calibration via `AcceptanceSensor.calibrate_baseline` → `_run_with_scheduler` (the single `RateScheduler` loop, attempt = `_adaptation`) → `_finalize_run` (`_after_run` → `_stabilize_at_full_rate`). `_maybe_characterize` (off unless `tuning_enable_characterization`) runs a non-destructive clone→probe→restore α-grid sweep whose `Profile` feeds the scheduler `epsilon`/policy (non-monotone → dense_grid safe mode). `_continue_to_full_rate` is retained as an `_after_run` helper for the rate-tuner families. **Relaxation-target anchor** (`_anchor_relaxation_target(baseline_val, real_target)`, default off → byte-identical): `run()` sets the missed-target relaxation anchor (`target_adjuster.target/original/floor`). Legacy = anchor on the rate-0 `baseline_val` (so on an ANN→LIF collapse the relaxation death-spirals to `baseline_val*(1-tol)`). With `tuning_target_floor_on_real_target` ON AND the real pipeline target (`pipeline.get_target_metric()` = the constructor's `target_accuracy`) ABOVE the collapsed baseline, it KEEPS the real-target anchor and caps the relaxation floor at `min(real, max(baseline_floor, real*(1-tol)))` — bounded below the real target, never down to the collapsed floor — so the model AIMS to recover toward the ANN. Falls back to baseline anchoring when the real target is unavailable / not above the baseline. The per-cycle rollback gate stays baseline-anchored (`acceptance_sensor.absolute_floor` untouched). Locked by `test_target_anchor.py`. **Bounded cosine stabilization** (`tuning_stabilization_bounded`, +`tuning_stabilization_ratio` default 0.5): when on, `_stabilize_at_full_rate` delegates to `_stabilize_bounded_cosine` — a SINGLE hard-cutoff pass of `int(ratio * _gradual_train_steps)` steps with a CosineAnnealingLR decay (chosen LR → ~0 over exactly N steps, via `train_steps_until_target(cosine_decay=True)`, unreachable target + `check_interval=N` so no early break), bracketed by the same pre/post rollback guard (non-destructive). Off → the existing patience/round-based pass (`_stabilization_budget` × `_max_stabilization_rounds`, cached LR) unchanged. **CERTIFIED non-destructive finalize guard** (`tuning_keepbest_certified`, R7d, default off → returns the finalize result unchanged): `_finalize_run` calls `_restore_best_committed_state` AFTER `_after_run` + `_stabilize_at_full_rate` + `_post_stabilization_hook` — a RUN-scope pre/post guard mirroring the local stabilization guard but bracketing the WHOLE ramp + forced jump to rate 1.0 + stabilization. When on, it re-reads the gate metric (`_certified_gate_metric`) of the finalized model and, iff it is worse than `_best_committed_metric` by more than `_rollback_tolerance`, restores `_best_committed_state` (snapshotted in `smooth_adaptation_cycle.py`) so the run can never ship below the best COMMITTED cycle. `run()` resets `_best_committed_state`/`_best_committed_metric` per run. `_log_cycle_summary` prints the `DecisionTrace` and (if the probe ran) the trend. **`_log_full_transform_trend`**: verdict keys on the GENUINE drop — CONVERGING when `last.genuine_drop < first.genuine_drop` (deployed cliff shrinking), else FLAT/DIVERGING. When genuine fails to shrink but `value_drop` DID, it warns "the value-domain probe WAS FOOLED" and surfaces the `proxy_gap` trajectory; the value drop alone is no longer a convergence signal. |
| `rate_tuner_seam.py` | `RateTunerSeam` (runtime-checkable `Protocol`), `RateTunerSeamMixin`, `OneShotRateTunerSeamMixin` | **E1 — the uniform rate-tuner seam.** The three driver-facing verbs every rate tuner exposes so an `OptimizationDriver` (`controller`/`fast`/…) drives ANY tuner generically: `ramp(rate)` (predictor — apply T at rate, return the post-apply metric), `recover_to(target)` (corrector — fine-tune toward an accuracy target), `probe()` (non-committing validation read), plus `seam_descriptor()`. The verbs DELEGATE to each tuner's CURRENT private methods (no behavior change, byte-identical): `RateTunerSeamMixin` (the `SmoothAdaptationTuner` family) maps `ramp`→`_update_and_evaluate`, `recover_to`→`_recover_to_target` (the shared recovery-engine SSOT the per-cycle `_recover` also calls), `probe`→`trainer.validate_n_batches`; `OneShotRateTunerSeamMixin` (the one-shot `ActivationShiftTuner`, which extends `TunerBase` not `SmoothAdaptationTuner`) gives the seam over its existing controller methods (`ramp`→`_axis.set_rate`, `recover_to`→`trainer.train_steps_until_target`, `probe`→`trainer.validate`). **EF2 (the E1↔E2 weld):** the seam is now driven from the run loop — the controller cycle (`smooth_adaptation_cycle.py`) routes its predictor through `ramp`, its reads (pre-cycle + post-recovery) through `probe`, and its corrector through `recover_to`; the fast-ladder rung (`fast_ladder.py`) reads its post-accuracy through `probe`. `recover_to` stashes the discovered LR on `self._last_recover_lr` so the cycle reads it back for the trace record while the public verb still returns the recovery RESULT (byte-identical — same SSOT primitive, same numbers). |
| `smooth_adaptation_tuner.py` | `SmoothAdaptationTuner` (= Seam+FastLadder+Cycle+Run+Base mixins), `UnifiedPerceptronTuner` (alias) | The composed orchestration tuner all rate tuners extend; mixes in `RateTunerSeamMixin` (the E1 seam) and `FastLadderMixin` (the E2 fast driver). `FastLadderMixin` precedes `SmoothAdaptationRunMixin` in the MRO so its `super().run()` / `super()._driver_attempt` / `super()._stabilization_budget` reach the controller path. |
| `fast_ladder.py` | `FastLadderMixin` | **E2 — the shared fast fixed-ladder optimization driver**, lifted out of `KDBlendAdaptationTuner` to the `SmoothAdaptationTuner` base so EVERY rate tuner (the analytical clamp/shift/activation-quant chain + the manager-rate family, which had NO fast path before) can be driven `controller` (default) or `fast`. Opt in via `_setup_fast_ladder(enabled, rates, steps_per_rate, eta_min_factor)` (normalizes `rates` to a trailing 1.0; default-off ⇒ `_fixed_ladder_policy` False ⇒ the unchanged controller loop ⇒ byte-identical). **EF1 (the CONSUMING half):** `_consume_optimization_driver(*, rates, steps_per_rate, eta_min_factor)` is the one call the analytical/manager families make in `__init__` to READ the pipeline-wide axis (`DeploymentPlan.of(pipeline).optimization_driver_for_family`), stash the resolved `OptimizationDriver` on `self._optimization_driver`, and configure the ladder from it — so they consume `DeploymentPlan.optimization_driver` instead of never reading the axis; default `controller` ⇒ `_setup_fast_ladder(enabled=False)` ⇒ byte-identical (`test_driver_axis_consumption.py`). When enabled: `run` resets the per-run fast scratch, `_driver_attempt`→`_fast_rate_attempt` (the `fixed_ladder` policy selected in `_run_with_scheduler`), `_stabilization_budget`→0. **EF2 (the E1↔E2 weld):** `_fast_rate_attempt` is now driven THROUGH the seam verbs — the fast PREDICTOR `_fast_ramp(rate)` (apply T via the uniform `_fast_set_rate`, then train the rung's `steps_per_rate` steps with ONE shared optimizer + spanning warmup/cosine LR via `_ensure_fast_optimizer`/`_build_fast_lr_schedule` + `_fast_loss`), then the universal `probe` verb reads the rung's post-accuracy (was an inlined `validate_n_batches`). `_fast_ramp` is the fast driver's analogue of the controller seam's `ramp`: the controller's `ramp` applies T + reads an instant metric because a SEARCH needs the read; the fixed schedule does not search per rung, so its predictor trains the rung in place and the one `probe` reads once. `_fast_set_rate` resolves the uniform setter (`_set_rate` → KD-blend/clamp/activation-adaptation, else `_apply_rate` → manager-rate). Records one `commit`/rung (`_record_fast_cycle`) and fires `_fast_probe`. `_fast_stabilize(steps, loss_fn=None)` is the post-finalize bounded recovery on the deployed forward (the fast driver's corrector, non-destructive). |
| `acceptance_sensor.py` | `AcceptanceSensor`, `BaselineRef` | The accept/reject/recovered decision service. Pure, bit-exact extraction of the loop's statistical math: `calibrate_baseline` (SE, empirical noise, clamped rollback tolerance, fixed baseline), `absolute_floor` (cumulative-drift guard), `is_catastrophic` (coarse pre-recovery fast-fail; `factor` is an injectable default = `CATASTROPHIC_DROP_FACTOR`, deliberately not SE-derived)/`rollback_threshold`/`ratchet_threshold`/`is_rollback`/`reached_target`. `ratchet_threshold(pre_cycle_acc, margin, best, cumulative_bound, floor)` is the NON-STALLING ratchet (`tuning_rollback_ratchet`): the MAX of the per-step relative bound (`pre_cycle_acc - margin`, so the ramp keeps climbing — no best-anchor stall), the cumulative-drift bound (`best - cumulative_bound`, caps total give-back below the best high-water mark, tightens as best ratchets up), and the absolute floor. **Paired McNemar gate (P2b)**: `paired_drop_se` / `paired_is_rollback` test the candidate vs the fixed-baseline correctness vector on a shared subsample (`(b10-b01)/N`, SE `sqrt(b10+b01)/N` — several-fold tighter than the marginal `0.5/sqrt(n)`). Wired into the rollback gate behind `tuning_use_paired_sensor` (off → marginal path bit-exact); Monte-Carlo-calibrated in `test_paired_sensor_calibration`. |
| `rate_scheduler.py` | `RateScheduler` | The single rate-search policy (spec §5.2 greedy-to-1.0 then bisect-the-gap) that replaced the legacy one-shot + grow/halve ramp + `_continue_to_full_rate` loops. The first (greedy/ladder) jump always runs; `epsilon` bounds only the bisection. Policies: `greedy_to_one` (default), `uniform_ladder` (KD-blend), `one_shot_only`, `dense_grid` (non-monotone safe mode — small uniform steps at the characterized `epsilon_hint`, never a greedy jump), `fixed_ladder` (schedule-not-search — walk an explicit `rates` list in order, commit each via `attempt`, never bisect; the home of the folded genuine-blend fast path). |
| `adaptation_driver.py` | `AdaptationDriver`, `CycleContext` | Orchestrator: `run()` drives the scheduler from `committed` toward 1.0 over a per-cycle `attempt`, then `finalize`. `build_scheduler` selects the policy (`policy_override` wins: `"fixed_ladder"` with an explicit `rates` list for a scheduled well-conditioned ramp, or `"dense_grid"` from a non-monotone characterization). `run_cycle(host, rate)` is the per-cycle predictor→corrector→commit/rollback **skeleton** dissolved out of the `_adaptation` god-method: it sequences the host's phase methods and branches on the `CycleContext` flags (catastrophic / rolled_back); decisions stay in `AcceptanceSensor`, snapshots in `CheckpointGuard`, recovery in `RecoveryEngine`. Building the driver standalone from the services is the remaining V6 polish. |
| `characterization.py` | `characterize`, `Profile`, `CalibrationSource`, `CascadeCharacterizer` | Pre-search profiling (spec §10 / V9): sweeps the drop on an α grid → `Profile(monotonic, max_slope, epsilon_hint, feasible_max, drops)`. A non-monotonicity beyond the noise budget (A1) triggers dense-grid safe mode; the steepest slope sets `epsilon_hint` (A3) so bisection never steps over a cliff. **Wired** into `_run_with_scheduler` via `_maybe_characterize` (behind `tuning_enable_characterization`, default off → bit-exact no-op). **E4 keystone activation — `CascadeCharacterizer`** is the REAL `conversion_policy.Characterizer` (subclass) for the cascaded fire-once cell: four forward-only float64 CPU probes — `cold_cascade_live` (live-depth fraction + output-layer liveness via the genuine single-spike cascade decoded through the segment forward's `node_value_recorder` side-channel), `ramp_monotone` (reuses `characterize`'s A1 verdict on the output decode swept over an S grid), `staircase_lif_ceiling` (the analytical cycle-accurate-OFF staircase output mean), `firing_gain` (deepest-layer genuine/staircase decoded ratio — collapses toward 0 on a pruned / dead-at-depth cascade). `characterize(model, recipe)` returns `CharacterizationResult(matches, probes, reason)` — matches iff the cold cascade is live, the ramp monotone, and the firing gain ≥ 0.1; an OFF-distribution model names the failing reason. `CalibrationSource` is the calibration-batch seam (`from_context(trainer)` pulls via `iter_validation_batches(n)`, `from_inputs(x)` for toy tests; `inputs()` returns the concatenated float64 batch or `None`). Tested in `test_conversion_policy_escalation.py`. |
| `recovery_engine.py` | `RecoveryEngine`, `PersistentOptimizerOwner`, `RESET_PER_CYCLE`, `PERSIST_WITHIN_CYCLE` | The corrector. `train_to_target` centralizes the `train_steps_until_target` + recovery-hook remove-in-`finally` pattern (per-cycle recovery, below-floor safety net, stabilization) and threads an optional owned `optimizer`. **Optimizer policy** (BAKED — no flag): the cycle's `_optimizer_policy()`/`_recovery_optimizer(lr)` build a `PersistentOptimizerOwner` optimizer reused across the LR sweep / recovery within a cycle so Adam moments survive (owner reset at `run()` start). Tuner families whose recovery replaces the parameter set each cycle set `_supports_persistent_optimizer = False` and pass `optimizer=None` (fresh build + del each call — bit-exact). LR discovery / tunable-param folding remain later extensions. |
| `ft_pass_wall.py` | `FtPassWallLog` | **AC5 — the per-fine-tuning-PASS wall log (timing-only, byte-identical).** AC5 = "no fine-tuning step exceeds 5 min" is judged PER fine-tuning PASS, NOT on the end-to-end pipeline wall (dominated by the non-FT Soft-Core-Mapping / Weight-Quantization / Simulation steps). `record(label, wall_s)` appends a `{label, wall_s}` entry (rejects a negative wall); `time_pass(label)` is the context-manager that times a block on `time.monotonic` (immune to wall-clock / NTP jumps; records the wall even when the block raises). `max_wall_s` is the worst single pass (0.0 when empty — AC5 reads a float, never `None`). The `SmoothAdaptationCycleMixin` owns one per run (`_ft_pass_wall_log()`, keyed to the `_run_t0` epoch so a re-run starts fresh) and times its fine-tuning passes into it; `max_ft_pass_wall_s` / `ft_pass_walls` / `ft_pass_wall_metrics()` surface the verdict number + breakdown that `chip_simulation.cost_extraction` carries on the `CostRecord` (`FT_PASS_WALLS_FILENAME`). Tested in `tests/unit/tuning/test_ft_pass_wall_instrumentation.py`. |
| `checkpoint_guard.py` | `CheckpointGuard`, `Handle` | Scoped, location-aware rollback snapshots. `scope=full`/`location=device` (default) delegates verbatim to `clone/restore_state_for_trainer` (byte-identical, golden-safe); `scope=tunable` clones only `requires_grad` params + buffers (skip frozen backbone); `location=cpu_pinned` does a **real** async offload — each device tensor → a fresh *pinned* host buffer via non-blocking d2h, one trailing sync; restore is non-blocking h2d + sync (frees ~1× model VRAM; ~3–8× faster than the old pageable path, measured on CUDA — `test_checkpoint_guard_cuda_benchmark`). The sole snapshot path in `_clone_state`/`_restore_state`; `scope`/`location` are config opt-ins defaulting to full/device. |
| `adaptation_manager.py` | `AdaptationManager` | Rate-field host (`clamp_rate`/`quantization_rate`/`shift_rate`/`noise_rate`/`activation_adaptation_rate`/`scale_rate`); `update_activation` rebuilds a perceptron's decorator stack from current rates. |
| `adaptation_manager_factory.py` | `create_adaptation_manager_for_model` | Build a manager + run initial `update_activation` for all perceptrons. |
| `tuning_budget.py` | `TuningBudget`, `tuning_budget_from_pipeline`, `max_total_training_steps`, `min_step_for_smooth_adaptation` | Step-based budget from dataset size; `accuracy_se()` = `0.5/sqrt(eval_sample_count)`. |
| `genuine_probe.py` | `eval_forward_over_val`, `genuine_acc_on_clone` | Pure helpers for the genuine full-transform probe. `eval_forward_over_val(trainer, forward_obj, model, n_batches, device)` = top-1 accuracy of an arbitrary `forward_obj` over `n_batches` of `trainer.iter_validation_batches` (never installs it on the model). `genuine_acc_on_clone(model, device, *, prepare, build_forward, evaluate)` runs the genuine eval on a `deepcopy` (prepare → build forward → evaluate), leaving the live model's state, activation structure, and installed `forward` untouched. | `ramp_strategy.py` | `RampStrategy`, `ValueDomainProxyRamp` | The polymorphic ramp strategy (how the model is driven 0→1) the KD-blend tuners delegate to — mirrors `spiking/segment_policies.py`. One cohesive class per ramp mode bundles the seams that vary together: `is_bare_target(tuner)`, `make_axis(tuner)`, `make_blend(tuner, old, target, rate)`, `ramp_forward(tuner, model)`, `make_kd_loss(tuner)` (delegates to the tuner's `_kd_classification_loss` SSOT, which reads the config-driven `kd_ce_alpha` / `kd_temperature` weighting of `α·CE + (1−α)·KD-to-ANN`; default 0.3 / 3.0 = the historical hardcode ⇒ byte-identical, shared by LIF + TTFS), `after_install_blend_pre(tuner)` (pre-ramp setup), `on_remove_forward(tuner)` (owned-reference cleanup). | `ttfs_adaptation_plan.py` | `TtfsAdaptationPlan` | Contract-driven resolution of the TTFS-cycle adaptation flag-thicket. `TtfsAdaptationPlan.resolve(config, *, synchronized, optimization_driver=None)` is the SINGLE place that reads the ~24 `ttfs_*` adaptation flags and applies the precedence / compatibility rules (genuine ramps cascaded-only; blend ⊐ annealed; STE ⇒ annealed and ⊥ blend; `ste_fast` ⊂ ste; `genuine_blend_fast` ⊂ blend; `proxy_fast` excludes the genuine bare-target ramps + synchronized; `ste_refine` ⊂ proxy_fast). **EF1:** the pipeline-wide `optimization_driver` axis is the controller-vs-fast GATE over the three-way fast fork — `_configure` passes `DeploymentPlan.of(pipeline).optimization_driver`, and an explicit `controller` vetoes every `*_fast` selector (the per-`ttfs_*`-flag still picks WHICH fast variant). `None` (back-compat for direct callers) derives the gate from the legacy fast flags via `resolve_optimization_driver(config)` ⇒ byte-identical. **EF3:** `resolve` also accepts two optional injected keystone seams (both default `None` ⇒ byte-identical) — `calibration_resolver` (the live tuner passes `contract.calibration_pipeline`, so the E3 conversion-health resolution is keyed by the (firing × sync) contract policy, not the tuner boolean; `None` ⇒ the raw `CalibrationPipeline.resolve` alias, bit-exact since the contract keys identically on cascaded-vs-synchronized) and `conversion_decision` (the E4 `ConversionDecision` from `ConversionPolicy.resolve`/`contract.conversion_policy`) — when the decision is `enabled` its `driver` OVERRIDES the axis gate (an escalation vetoes the fast fork exactly like an explicit `controller`) AND its recipe's `train_s_hint` is threaded onto `plan.train_s_hint` (R2's two-residual decouple — the tuner trains the genuine cascade at train_S, deploys at deploy_S); default-off ⇒ the decision is inert (`enabled=False` ⇒ `train_s_hint=None`) ⇒ the axis is untouched and the S is the configured one ⇒ byte-identical. It **composes the adaptation trifecta**: the ramp-strategy flags (above), the `OptimizationDriver` (the controller-vs-fast-ladder rung, `plan.driver`), and the `CalibrationPipeline` (the conversion-health steps, `plan.calibration`, E3-keyed by the (firing × sync) `SpikingModePolicy` — cascaded cycle opts in, synchronized cycle inert; distmatch driven by the blend ramp). The legacy `fast_ladder_*` fields are now read-only properties mirroring `plan.driver` (back-compat). Returns a frozen plan; `TTFSCycleAdaptationTuner._configure` binds it to its `self._*` fields, so the dispatch is read in one tested place (`test_ttfs_adaptation_plan.py`) instead of re-derived inline. |
| `optimization_driver.py` | `OptimizationDriver` | The optimization-driver concern (P3 / E2): *how* the rate is driven 0→1, orthogonal to the ramp strategy. Two resolvers: (1) `OptimizationDriver.for_family(*, fast, rates, steps_per_rate, eta_min_factor)` — the GENERIC `controller | fast` axis for the single-switch families (LIF + the analytical clamp/shift/quant/manager-rate chain); one declarative switch, `fast=False` ⇒ controller ⇒ byte-identical. (2) `OptimizationDriver.resolve(*, staircase_ste_fast, genuine_blend_fast, proxy_fast, ste_steps, blend_fast_rates, blend_fast_steps_per_rate, blend_fast_lr_eta_min)` — the richer TTFS resolver whose fast path forks three ways.
