# tuning/orchestration/ -- Smooth-adaptation orchestration loop + services

The control core of the tuning subsystem: the rate-search loop, the per-cycle
predictor→corrector→commit/rollback machinery, the adaptation-manager rate host,
and the decision services the refactor is extracting from the loop.

## Key Components

| File | Symbols | Purpose |
|------|---------|---------|
| `tuner_base.py` | `TunerBase`, `CATASTROPHIC_DROP_FACTOR`, `_RECOVERY_PATIENCE`, `_STUCK_STREAK_REQUIRED` | Shared infrastructure: trainer, budget, target adjuster, LR finder (`_find_lr`/`_get_cached_lr` with `anchor_lr=pipeline_lr`), `validate`. Tuning constants. `_create_trainer` builds the `BasicTrainer` with `recipe=self._tuning_recipe()` (the `tuning_recipe` block) and forwards `tuning_recipe_recovery` (config flag, default off → byte-identical) so the STEP recovery honors the recipe optimizer + warmup/cosine schedule instead of the hardcoded Adam/constant-LR path. |
| `smooth_adaptation_cycle.py` | `SmoothAdaptationCycleMixin` | One adaptation cycle, now **dissolved**: `_adaptation` is a 1-line delegate to `AdaptationDriver.run_cycle`, which drives the named phase methods here — `_begin_cycle` (clone pre-state + pre-cycle eval), `_probe_instant` (`_update_and_evaluate` + catastrophic gate), `_recover` (LR + recovery train), `_measure_post` (post eval + dual rollback gate), `_rollback_cycle` / `_commit_cycle` (restore-or-commit + stuck-streak target relaxation). Records one `DecisionRecord` per exit into `self._cycle_log`. Accept/reject decisions delegate to `AcceptanceSensor` (static methods); `_absolute_post_acc_floor` delegates too. Under `tuning_use_paired_sensor`, `_measure_post` derives `post_acc` from the paired correctness vector itself (one eval pass, one statistical basis — D6); a non-negative `global_budget` is enforced at construction (D4). **Recovery-quality knobs** (all default-off → byte-identical): (1) `tuning_refind_lr_on_miss` invalidates the cached LR when a committed cycle misses the target (`_commit_cycle`) so the next cycle re-discovers it (else the LR is cached at cycle 0 forever); (2) `tuning_recovery_lr_plateau` (+`_factor`/`_reductions`, `_recovery_plateau_kwargs`) passes plateau LR-reduction kwargs to `_recover`'s `RecoveryEngine.train_to_target` (off → 1.0/0 no-op); (3) `tuning_rollback_ratchet` (+`tuning_rollback_cumulative_bound`, default 0.05) is the NON-STALLING ratchet in `_measure_post`: `AcceptanceSensor.ratchet_threshold` keeps the per-step relative gate (`pre_cycle_acc - margin`, so the ramp keeps climbing) but caps the cumulative drift below the `_best_committed_acc` high-water mark (`_ratchet_best_anchor`, tightened on each commit by `_update_best_committed_acc`) — repeated sub-margin slips cannot accumulate and the bound tightens as best ratchets up; (4) `tuning_tight_plateau` (+`tuning_recovery_check_divisor`, `_recovery_check_interval`) divides the recovery check interval so the stale-streak patience trips after fewer steps; (5) `tuning_stabilization_bounded` makes `_recover` request `return_steps` and `_accumulate_gradual_steps` sums them into `_gradual_train_steps` (the bounded-stabilization budget, consumed in `smooth_adaptation_run.py`). **GENUINE full-transform probe** (`tuning_full_transform_probe`, default off): after each commit, `_probe_full_transform` measures BOTH the value-domain (`_value_full_transform_eval`, always non-destructive clone→rate-1.0→restore) AND the genuine deployed-forward (`_full_transform_eval`, base default = value; overridden by `KDBlendAdaptationTuner`) rate-1.0 accuracy. Logs a per-commit record `{committed, committed_acc, value_full_acc, genuine_full_acc, value_drop, genuine_drop, proxy_gap}` (with `full_acc`/`drop` aliasing the value pair for the legacy test). `proxy_gap = value_full_acc − genuine_full_acc` is the value↔genuine divergence the value proxy hides. The verdict (`_log_full_transform_trend`, in `smooth_adaptation_run.py`) keys on `genuine_drop`. |
| `smooth_adaptation_run.py` | `SmoothAdaptationRunMixin` | `run()`: baseline calibration via `AcceptanceSensor.calibrate_baseline` → `_run_with_scheduler` (the single `RateScheduler` loop, attempt = `_adaptation`) → `_finalize_run` (`_after_run` → `_stabilize_at_full_rate`). `_maybe_characterize` (off unless `tuning_enable_characterization`) runs a non-destructive clone→probe→restore α-grid sweep whose `Profile` feeds the scheduler `epsilon`/policy (non-monotone → dense_grid safe mode). `_continue_to_full_rate` is retained as an `_after_run` helper for the rate-tuner families. **Bounded cosine stabilization** (`tuning_stabilization_bounded`, +`tuning_stabilization_ratio` default 0.5): when on, `_stabilize_at_full_rate` delegates to `_stabilize_bounded_cosine` — a SINGLE hard-cutoff pass of `int(ratio * _gradual_train_steps)` steps with a CosineAnnealingLR decay (chosen LR → ~0 over exactly N steps, via `train_steps_until_target(cosine_decay=True)`, unreachable target + `check_interval=N` so no early break), bracketed by the same pre/post rollback guard (non-destructive). Off → the existing patience/round-based pass (`_stabilization_budget` × `_max_stabilization_rounds`, cached LR) unchanged. `_log_cycle_summary` prints the `DecisionTrace` and (if the probe ran) the trend. **`_log_full_transform_trend`**: verdict keys on the GENUINE drop — CONVERGING when `last.genuine_drop < first.genuine_drop` (deployed cliff shrinking), else FLAT/DIVERGING. When genuine fails to shrink but `value_drop` DID, it warns "the value-domain probe WAS FOOLED" and surfaces the `proxy_gap` trajectory; the value drop alone is no longer a convergence signal. |
| `smooth_adaptation_tuner.py` | `SmoothAdaptationTuner` (= Cycle+Run+Base mixins), `UnifiedPerceptronTuner` (alias) | The composed orchestration tuner all rate tuners extend. |
| `acceptance_sensor.py` | `AcceptanceSensor`, `BaselineRef` | The accept/reject/recovered decision service. Pure, bit-exact extraction of the loop's statistical math: `calibrate_baseline` (SE, empirical noise, clamped rollback tolerance, fixed baseline), `absolute_floor` (cumulative-drift guard), `is_catastrophic` (coarse pre-recovery fast-fail; `factor` is an injectable default = `CATASTROPHIC_DROP_FACTOR`, deliberately not SE-derived)/`rollback_threshold`/`ratchet_threshold`/`is_rollback`/`reached_target`. `ratchet_threshold(pre_cycle_acc, margin, best, cumulative_bound, floor)` is the NON-STALLING ratchet (`tuning_rollback_ratchet`): the MAX of the per-step relative bound (`pre_cycle_acc - margin`, so the ramp keeps climbing — no best-anchor stall), the cumulative-drift bound (`best - cumulative_bound`, caps total give-back below the best high-water mark, tightens as best ratchets up), and the absolute floor. **Paired McNemar gate (P2b)**: `paired_drop_se` / `paired_is_rollback` test the candidate vs the fixed-baseline correctness vector on a shared subsample (`(b10-b01)/N`, SE `sqrt(b10+b01)/N` — several-fold tighter than the marginal `0.5/sqrt(n)`). Wired into the rollback gate behind `tuning_use_paired_sensor` (off → marginal path bit-exact); Monte-Carlo-calibrated in `test_paired_sensor_calibration`. |
| `rate_scheduler.py` | `RateScheduler` | The single rate-search policy (spec §5.2 greedy-to-1.0 then bisect-the-gap) that replaced the legacy one-shot + grow/halve ramp + `_continue_to_full_rate` loops. The first (greedy/ladder) jump always runs; `epsilon` bounds only the bisection. Policies: `greedy_to_one` (default), `uniform_ladder` (KD-blend), `one_shot_only`, `dense_grid` (non-monotone safe mode — small uniform steps at the characterized `epsilon_hint`, never a greedy jump), `fixed_ladder` (schedule-not-search — walk an explicit `rates` list in order, commit each via `attempt`, never bisect; the home of the folded genuine-blend fast path). |
| `adaptation_driver.py` | `AdaptationDriver`, `CycleContext` | Orchestrator: `run()` drives the scheduler from `committed` toward 1.0 over a per-cycle `attempt`, then `finalize`. `build_scheduler` selects the policy (`policy_override` wins: `"fixed_ladder"` with an explicit `rates` list for a scheduled well-conditioned ramp, or `"dense_grid"` from a non-monotone characterization). `run_cycle(host, rate)` is the per-cycle predictor→corrector→commit/rollback **skeleton** dissolved out of the `_adaptation` god-method: it sequences the host's phase methods and branches on the `CycleContext` flags (catastrophic / rolled_back); decisions stay in `AcceptanceSensor`, snapshots in `CheckpointGuard`, recovery in `RecoveryEngine`. Building the driver standalone from the services is the remaining V6 polish. |
| `characterization.py` | `characterize`, `Profile` | Pre-search profiling (spec §10 / V9): sweeps the drop on an α grid → `Profile(monotonic, max_slope, epsilon_hint, feasible_max, drops)`. A non-monotonicity beyond the noise budget (A1) triggers dense-grid safe mode; the steepest slope sets `epsilon_hint` (A3) so bisection never steps over a cliff. **Wired** into `_run_with_scheduler` via `_maybe_characterize` (behind `tuning_enable_characterization`, default off → bit-exact no-op). |
| `recovery_engine.py` | `RecoveryEngine`, `PersistentOptimizerOwner`, `RESET_PER_CYCLE`, `PERSIST_WITHIN_CYCLE` | The corrector. `train_to_target` centralizes the `train_steps_until_target` + recovery-hook remove-in-`finally` pattern (per-cycle recovery, below-floor safety net, stabilization) and threads an optional owned `optimizer`. **Optimizer policy** (BAKED — no flag): the cycle's `_optimizer_policy()`/`_recovery_optimizer(lr)` build a `PersistentOptimizerOwner` optimizer reused across the LR sweep / recovery within a cycle so Adam moments survive (owner reset at `run()` start). Tuner families whose recovery replaces the parameter set each cycle set `_supports_persistent_optimizer = False` and pass `optimizer=None` (fresh build + del each call — bit-exact). LR discovery / tunable-param folding remain later extensions. |
| `checkpoint_guard.py` | `CheckpointGuard`, `Handle` | Scoped, location-aware rollback snapshots. `scope=full`/`location=device` (default) delegates verbatim to `clone/restore_state_for_trainer` (byte-identical, golden-safe); `scope=tunable` clones only `requires_grad` params + buffers (skip frozen backbone); `location=cpu_pinned` does a **real** async offload — each device tensor → a fresh *pinned* host buffer via non-blocking d2h, one trailing sync; restore is non-blocking h2d + sync (frees ~1× model VRAM; ~3–8× faster than the old pageable path, measured on CUDA — `test_checkpoint_guard_cuda_benchmark`). The sole snapshot path in `_clone_state`/`_restore_state`; `scope`/`location` are config opt-ins defaulting to full/device. |
| `adaptation_manager.py` | `AdaptationManager` | Rate-field host (`clamp_rate`/`quantization_rate`/`shift_rate`/`noise_rate`/`activation_adaptation_rate`/`scale_rate`); `update_activation` rebuilds a perceptron's decorator stack from current rates. |
| `adaptation_manager_factory.py` | `create_adaptation_manager_for_model` | Build a manager + run initial `update_activation` for all perceptrons. |
| `tuning_budget.py` | `TuningBudget`, `tuning_budget_from_pipeline`, `max_total_training_steps`, `min_step_for_smooth_adaptation` | Step-based budget from dataset size; `accuracy_se()` = `0.5/sqrt(eval_sample_count)`. |
| `genuine_probe.py` | `eval_forward_over_val`, `genuine_acc_on_clone` | Pure helpers for the genuine full-transform probe. `eval_forward_over_val(trainer, forward_obj, model, n_batches, device)` = top-1 accuracy of an arbitrary `forward_obj` over `n_batches` of `trainer.iter_validation_batches` (never installs it on the model). `genuine_acc_on_clone(model, device, *, prepare, build_forward, evaluate)` runs the genuine eval on a `deepcopy` (prepare → build forward → evaluate), leaving the live model's state, activation structure, and installed `forward` untouched. Both are callable-driven (no tuner imports) so they stay testable in isolation. |
| `ramp_strategy.py` | `RampStrategy`, `ValueDomainProxyRamp` | The polymorphic ramp strategy (how the model is driven 0→1) the KD-blend tuners delegate to — mirrors `spiking/segment_policies.py`. One cohesive class per ramp mode bundles the seams that vary together: `is_bare_target(tuner)`, `make_axis(tuner)`, `make_blend(tuner, old, target, rate)`, `ramp_forward(tuner, model)`, `make_kd_loss(tuner)`, `after_install_blend_pre(tuner)` (pre-ramp setup), `on_remove_forward(tuner)` (owned-reference cleanup). The base returns the value-domain proxy behavior; `ValueDomainProxyRamp` is the default (LIF + the proxy TTFS ramp). The genuine/STE TTFS strategies live next to their forwards/losses in `tuners/ttfs_cycle_adaptation_tuner.py`, so this module imports only `axes.BlendAxis` (deferred local imports otherwise) and stays a tuner-imported leaf. |
| `ttfs_adaptation_plan.py` | `TtfsAdaptationPlan` | Contract-driven resolution of the TTFS-cycle adaptation flag-thicket. `TtfsAdaptationPlan.resolve(config, *, synchronized)` is the SINGLE place that reads the ~24 `ttfs_*` adaptation flags and applies the precedence / compatibility rules (genuine ramps cascaded-only; blend ⊐ annealed; STE ⇒ annealed and ⊥ blend; `ste_fast` ⊂ ste; `genuine_blend_fast` ⊂ blend; `proxy_fast` excludes the genuine bare-target ramps + synchronized; `ste_refine` ⊂ proxy_fast). It **composes the adaptation trifecta**: the ramp-strategy flags (above), the `OptimizationDriver` (the controller-vs-fast-ladder rung, `plan.driver`), and the `CalibrationPipeline` (the conversion-health steps, `plan.calibration`, distmatch driven by the blend ramp). The legacy `fast_ladder_*` fields are now read-only properties mirroring `plan.driver` (back-compat). Returns a frozen plan; `TTFSCycleAdaptationTuner._configure` binds it to its `self._*` fields, so the dispatch is read in one tested place (`test_ttfs_adaptation_plan.py`) instead of re-derived inline. |
| `optimization_driver.py` | `OptimizationDriver` | The optimization-driver concern (P3): *how* the rate is driven 0→1, orthogonal to the ramp strategy. `OptimizationDriver.resolve(*, staircase_ste_fast, genuine_blend_fast, proxy_fast, ste_steps, blend_fast_rates, blend_fast_steps_per_rate, blend_fast_lr_eta_min)` maps the (already precedence-resolved) fast selectors to the concrete `_setup_fast_ladder` rung — `controller` (default, full SmoothAdaptation machinery) vs `fast_ladder` (the schedule-not-search `fixed_ladder` policy). STE-fast → one rung at rate 1.0 (its dedicated split-LR + progressive-depth loop runs the steps); blend/proxy → the multi-rung ladder, the proxy flooring the endpoint LR (value-domain endpoint needs real recovery; the genuine-blend's genuine-CE carries it). Tested in `test_optimization_driver.py`. |
| `calibration_pipeline.py` | `CalibrationPipeline` | The conversion-health concern (P2): the composable calibration steps (gain-correction / theta-cotrain / distmatch / boundary-STE) + their compatibility. `CalibrationPipeline.resolve(config, *, synchronized, distmatch_driven)` resolves which steps are active and applies the explicit `compatible_with` rules formerly scattered as per-flag guards in the TTFS tuner `_configure`: every step cascaded-only (off when synchronized); gain-correction has two mutually-exclusive modes with the RAMP winning over the COLD trim; theta-cotrain is compatible_with everything *except* the gain RAMP (both manage `activation_scale`, gain wins); distmatch is active iff the genuine-blend ramp drives it. Carries the gain (`gain_rule`/`gain_c`) and distmatch (`distmatch_quantile`/`distmatch_bias_iters`/`distmatch_bias_eta`) numeric params + the boundary-STE temperature (`boundary_surrogate_temp`, None = severed backward). The tuner binds the resolved decisions; the parity-critical *application* order stays in the tuner. Tested in `test_calibration_pipeline.py`. |
| `kd_blend_adaptation_tuner.py` | `KDBlendAdaptationTuner`, `BlendActivation`, `_InstalledForward` | ANN→SNN blend ramp (LIF/TTFS base): live `BlendActivation.rate` via `set_blend_rate`, KD recovery against a frozen teacher, parity-critical `_finalize` forward-install. **`self._ramp` (a `RampStrategy`, built in `__init__` from `_make_ramp_strategy()` right after `_configure`, before the seams run)** owns the 0→1 dispatch: `_make_blend`/`_make_axis`/`_make_kd_loss`/`_ramp_forward` delegate to it, `_after_install_blend` runs `_ramp.after_install_blend_pre` then `_install_ramp_forward`, and `_remove_forward` calls `_ramp.on_remove_forward` before unpatching. Default `_make_ramp_strategy()` → `ValueDomainProxyRamp` (LIF inherits it). **Shared genuine-forward builder `_finalize_forward_for(model)`** (default `None`; tuners override — ttfs cascaded → `_SegmentSpikeForward(model, T)`, cycle-accurate LIF → its NF forward): `_finalize_forward` and the finalize install both route through it, so the probed forward ≡ the deployed forward. **`_finalize_rebuild(model)`** is the model-parameterized finalize-style activation rebuild (`_before_/_after_finalize_rebuild` hooks default to `self.model`), shared by `_finalize` (live model) and the probe (isolated clone) so probe ≡ deploy. The overridden **`_full_transform_eval`** runs that deployed forward at rate 1.0 on a deepcopy via `genuine_acc_on_clone` (clone → `set_blend_rate(1)` → `_finalize_rebuild` → build forward → `eval_forward_over_val`), snapshotting/restoring `_finalize_manager_flags` so the shared `adaptation_manager` survives; it falls back to the value-domain measure when the builder is `None`. **`_make_axis()`** delegates to `self._ramp.make_axis(self)` (default `BlendAxis`; the TTFS strategies pick `TTFSGenuineAxis`/`GenuineBlendAxis`). `_after_run` records `finalize_cliff = ramp_metric − post_finalize` and `_report_cliff_probe_consistency` warns if the last per-commit `genuine_drop` and `finalize_cliff` (same gap, different anchor) diverge > 0.1. **Shared fast fixed-ladder machinery** (`_setup_fast_ladder`, `_fast_rate_attempt`, `_ensure_fast_optimizer`, `_record_fast_cycle`, `_build_fast_lr_schedule`, the `run`-reset / `_driver_attempt`-routing / `_stabilization_budget`→0): the schedule-not-search fast ramp shared by LIF + TTFS (subclasses opt in via `_setup_fast_ladder`), with `_fast_loss`/`_fast_probe` hooks (default `_fast_loss` = the installed KD loss; TTFS overrides to plain-CE+genuine-CE for its genuine blend), `_fast_eta_min_factor` (floors the spanning-cosine endpoint LR so a value-domain endpoint keeps recovering), and **`_fast_stabilize(steps)`** (a non-destructive post-finalize bounded recovery on the DEPLOYED forward — for value-domain ramps, LIF + the TTFS proxy, whose endpoint needs the deployed dynamics trained; the genuine-blend ramp trains through the deployed cascade so it skips this). |
