# scripts/campaign/ -- Autonomous Research Campaign Daemons

Long-lived daemons that keep the GPU queue full and the research loop turning
without a human in the loop. They produce/consume `scripts/gpu/gpu_queue.py` jobs.

## Key Components

| File | Symbols | Purpose |
|------|---------|---------|
| `scheduler.py` | `Scheduler`, `instantiate`, `onchip_precheck`, `_classify_cfg_validity`, `capacity_precheck`, `_estimate_cfg_capacity`, `set_path`, `get_path`, `existing_ids` | FILLS the queue from a declarative `backlog.json`: instantiates each batch's config grid, dedupes against everything enqueued/run, refills to a high-watermark. Before enqueuing, runs the TIERED validity pre-check (`onchip_precheck` → `classify_validity`, both params AND macs): only an INVALID model (`min(param,mac)` on-chip below `deployment_parameters.onchip_min_fraction`, default 0.20) is SKIPPED/logged `invalid_host_majority` so it never claims a GPU. VALID and VALID_FLAGGED are ADMITTED; a flagged job (below the 0.50 majority `onchip_majority_fraction`) is logged with its named `research_gap_ops`/`placement_fixable_ops` as transferable-tuning evidence. It THEN runs the E4 capacity pre-check (`capacity_precheck` → `_estimate_cfg_capacity` → `estimate_cores_needed`): a config whose STATIC hard-core lower bound exceeds its declared core budget is SKIPPED/logged `capacity_exceeded` (naming `cores_needed`/`cores_available`/`overflowing_segment`) so an unplaceable net (VGG16@224 on a 1000-core budget) never claims a GPU. The pre-check is scheduling-aware: when the chip permits `allow_scheduling`, a net too big for one pool is ADMITTED as feasible-via-scheduling whenever its PEAK reprogram phase fits (the `info` carries `scheduled`/`peak_phase_cores`/`phase_count`); only a peak-phase or single-atomic-bundle overflow is rejected. Gated by `onchip_majority_gate` / `capacity_gate`; model-build / IR / classification / estimate failures are NON-FATAL — the job is enqueued anyway. Optionally acquires a `guards.singleton_lock('scheduler')` so only one instance runs — OPT-IN via `--singleton` or `CAMPAIGN_SINGLETON=1`; the default invocation takes no lock (byte-identical). |
| `director.py` | research director | GROWS the backlog from ledger findings + FLAGS uncovered runs. |
| `experiment_matrix.py` | `gen_f1_batches`, `gen_f2_batches`, `gen_f3_batches`, `gen_all_batches`, `default_matrix`, `f3_dual_regime_matrix`, `cifar_breadth_matrix`, `gen_cifar_breadth_batches`, `emit_cifar_breadth_backlog`, `CIFAR_PROVIDERS`, `PRETRAINED_WEIGHT_SOURCE`, `MatrixCell`, `validate_batch`, `BatchSchemaError`, `read_ledger`, `aggregate_f1`, `aggregate_f2`, `aggregate_f3`, `write_findings_markdown`, `render_findings_markdown`, `T_CRIT_95`, `main` (CLI) | **F-harness — the publication experiment matrix (no GPU).** Two halves. (1) BATCH GENERATORS emit `backlog.json`-shaped batch dicts (the shape `scheduler.instantiate`/`Scheduler.refill` consume) for three studies: F1 (CIs + ablations — multi-seed cells across the honestly-priced matrix), F2 (baseline head-to-head — percentile-norm `activation_scale_quantile=1.0` vs the default `0.99` clip on each covered/valid cell), F3 (dual-regime — `from_scratch` vs `pretrained` via `preload_weights` per (model,dataset)). **F3 needs a vehicle with a REAL pretrained source**: native `default_matrix()` cells (deep_cnn/lenet5) have no `get_pretrained_factory()`, so `preload_weights=True` is ill-posed (crashed rc=1); F3 runs on `f3_dual_regime_matrix()` — `torch_squeezenet11` on `CIFAR10_DataProvider` (template `cifar_squeezenet11_dualregime.json`, same arch both arms). `gen_f3_batches` SKIPS cells with `MatrixCell.pretrained_source` `None` and REJECTS a non-`PRETRAINED_WEIGHT_SOURCE` (checkpoint) source for the boolean grid. **B2 CIFAR breadth** (`cifar_breadth_matrix` / `gen_cifar_breadth_batches`, study tag `B2`) extends the dataset-margin law to RGB: deep_cnn (cascaded+synchronized, d=4/6/8) and lenet5 (synchronized) on `CIFAR10_DataProvider`/`CIFAR100_DataProvider`, reusing the MNIST deep_cnn/lenet5 templates and swapping the dataset through the grid's `data_provider_name` axis (no new template); both models are VALID on CIFAR at 3×32×32 (param/MAC fractions clear the 0.50 majority). `emit_cifar_breadth_backlog(out_path)` writes the 4 breadth batches (42 job-cells) to a JSON FILE only — the orchestrator appends to the live backlog, this module never does. Every emitted batch is `validate_batch`-checked against the backlog schema (required `id`/`template`/`grid`/`id_template`, non-empty grid axes, `id_template` fields ⊆ grid leaves) and defaults `enabled:false` (inert until the director rolls it out — default-off). (2) AGGREGATOR reads `runs/campaign/ledger.jsonl` in TWO schemas. (a) **Study-tagged per-seed rows** (`study:F1|F2|F3` + flat `deployed_acc`, the generators' write-back target) → per-cell mean +/- 95% t-CI (`source:per_seed`; `n<2`→0 half-width, scipy-free via the embedded `T_CRIT_95` table + 1.96 normal-approx for `df≥30`). (b) **Pre-aggregated SUMMARY rows** — what the LIVE campaign actually writes: ONE row per cell, run_ids prefixed `f1_`/`f2_`/`f3_` (the only F-study membership signal — `_summary_study`), both schedule arms in one row (`{schedule}_deployed_mean`/`_std_pp`, the F2 `quantile_head_to_head` row carries `_q099`/`_q10` arms, the F3 `mode` row carries the from_scratch arm's REAL `{schedule}_deployed_full_test_scm_per_seed` lists). Each summary row emits one cell PER schedule arm (`source:summary`) carrying the row's mean + `std_pp` with the 95% CI marked UNAVAILABLE (`ci_available:False`, `ci95:None` — a std is NEVER reshaped into a CI), plus `ann_test_acc_mean`/validity/verdict; a crashed/0-seed arm (no mean, e.g. F1 fmnist d6, F3 pretrained `dual_regime_missing_arms`) is emitted as an explicit NOT-MEASURED cell (`measured:False`/`delta_measured:False`), never silently dropped. Emits the F1 CI table, the F2 head-to-head delta table (percentile − default, pp), and the F3 dual-regime delta table (pretrained − from_scratch, pp); `write_findings_markdown` renders all three (with a `source` column + NOT-MEASURED / CI-unavailable markers) into `docs/research/findings/F_experiment_matrix.md`. CLI: `generate --study {f1,f2,f3,all,cifar}` / `aggregate --ledger … --out …`. The runs are enqueued/run later by the scheduler+runner — this module never touches a GPU. |
| `research_loop.py` | research-loop primitives | Enqueue/wait/results/ledger helpers for research workflows. |
| `coverage_report.py` | `main` (CLI) | **Frontier E1 CLI** — prints the hypervolume coverage report over `runs/campaign/ledger.jsonl` (`--ledger`): the tier tally (VALID / VALID_FLAGGED / INVALID), the coverage FRACTION against a claimed sub-product (`--vehicle` / `--dataset` / `--sync` / `--firing` / `--S` pin axes), the named UNTESTED frontier, the RESEARCH-GAP and PLACEMENT-FIXABLE frontiers, and (`--axes`) the orthogonal-vs-interacting axis classification. `--json` emits the machine-readable report. Delegates to `mimarsinan.chip_simulation.coverage_ledger`. |
| `guards.py` | `assert_base_current`, `snapshot_stash_list`, `assert_stash_intact`, `singleton_lock`, `StaleBaseError`, `StashTamperError`, `AlreadyRunningError` | **A4 engineering self-defense** — guards the three traps the automation has hit. `assert_base_current` raises `StaleBaseError` unless HEAD is `main` or a descendant (`git merge-base --is-ancestor`) — the stale-base trap. `snapshot_stash_list` captures stash entries by their immutable commit SHA (`--format=%H %gs`, so a freshly pushed stash never renumbers an existing one) and `assert_stash_intact` raises `StashTamperError` if any snapshotted entry vanished or changed — the stash-pop trap. `singleton_lock(name, lockdir=…)` is an `fcntl.flock` pidfile context manager (default lockdir `runs/campaign/locks/`) that raises `AlreadyRunningError` on a second acquire and releases on exit — the double-run trap. |

## Dependencies

- **Internal (lazy)**: `mimarsinan.mapping.verification.onchip_fraction.classify_validity`, `mimarsinan.pipelining.core.registry.model_registry.ModelRegistry`, `mimarsinan.data_handling.data_provider_factory.BasicDataProviderFactory` — imported only inside `_classify_cfg_validity` so importing the scheduler stays light for the daemon. The E4 capacity pre-check `_estimate_cfg_capacity` additionally (lazily) imports `mimarsinan.config_schema.runtime.build_flat_pipeline_config`, `mimarsinan.torch_mapping.convert_torch_model`, `mimarsinan.mapping.ir_mapping_class.IRMapping`, and `mimarsinan.mapping.verification.capacity.estimate_cores_needed` to build the IR and run the static core-count bound. `coverage_report.py` imports `mimarsinan.chip_simulation.coverage_ledger` (the E1 coverage API).
- **External**: `scripts/gpu/gpu_queue.GpuQueue`. `guards.py` depends only on the stdlib (`subprocess`, `fcntl`, `contextlib`) and the `git` CLI; `scheduler.py` imports `guards.singleton_lock` lazily, only on the opt-in `--singleton` / `CAMPAIGN_SINGLETON=1` path. `test_guards.py` covers the guards against hermetic throwaway `git init` repos.

## Dependents

- `campaign_runner` (in `scripts/gpu/`) DRAINS the queue the scheduler fills.
